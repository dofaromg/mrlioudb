## 部署前準備

### 1. 必要條件

#### GCP 帳號和專案
- ✅ 擁有 Google Cloud Platform 帳號
- ✅ 已建立 GCP 專案（預設: `flowmemorysync`）
- ✅ 已啟用計費
- ✅ 擁有必要的 IAM 權限:
  - `roles/container.admin` (GKE 管理)
  - `roles/iam.serviceAccountUser`
  - `roles/artifactregistry.admin` (容器映像管理)

#### 本地工具
```bash
# 檢查必要工具
gcloud --version
kubectl version --client
kustomize version
```

如果缺少工具，請安裝:
```bash
# 安裝 gcloud CLI
curl https://sdk.cloud.google.com | bash
exec -l $SHELL

# 安裝 kubectl
gcloud components install kubectl

# 安裝 kustomize
curl -s "https://raw.githubusercontent.com/kubernetes-sigs/kustomize/master/hack/install_kustomize.sh" | bash
sudo mv kustomize /usr/local/bin/
```

### 2. 配置檢查

運行配置驗證腳本:
```bash
bash scripts/validate_deployment.sh
```

預期輸出應包含:
- ✅ 工具檢查完成
- ✅ YAML 語法驗證完成
- ✅ Kustomize 建置成功
- ✅ 所有驗證通過

### 3. 環境變數設定

```bash
export PROJECT_ID=flowmemorysync
export REGION=asia-east1
export ZONE=asia-east1-a
export CLUSTER_NAME=modular-cluster
export NS=flowagent
```

**重要**: 如果你使用不同的專案，請修改這些值。

## 快速部署

### 選項 A: 一鍵部署 (推薦)

使用自動化部署腳本:

```bash
# 1. 克隆 repository (如果還沒有)
git clone https://github.com/dofaromg/flow-tasks.git
cd flow-tasks

# 2. 執行實際部署腳本
bash scripts/actual_deploy.sh
```

腳本將自動執行:
1. 檢查前置需求
2. 驗證配置文件
3. 初始化 GKE 叢集
4. 建立命名空間
5. 部署應用程式
6. 部署監控系統
7. 等待部署完成
8. 驗證部署狀態
9. 獲取訪問資訊

### 選項 B: 分步驟手動部署

如果你想要更多控制，可以分步驟執行:

```bash
# 1. 初始化 GKE 叢集
bash scripts/oneclick_gke_init.sh

# 2. 部署應用程式
kubectl apply -k cluster/overlays/prod/

# 3. 部署監控 (可選)
kubectl apply -k cluster/overlays/monitoring/

# 4. 檢查狀態
bash scripts/check_deployment_status.sh
```

## 詳細部署步驟

### 步驟 1: 設定 GCP 專案

```bash
# 設定預設專案
gcloud config set project $PROJECT_ID

# 啟用必要的 API
gcloud services enable container.googleapis.com
gcloud services enable artifactregistry.googleapis.com
gcloud services enable compute.googleapis.com
```

### 步驟 2: 建立或連接 GKE 叢集

#### 2.1 如果叢集已存在

```bash
# 取得叢集憑證
gcloud container clusters get-credentials $CLUSTER_NAME \
  --zone $ZONE \
  --project $PROJECT_ID
```

#### 2.2 如果需要建立新叢集

```bash
# 建立 GKE 叢集 (約需 5-10 分鐘)
gcloud container clusters create $CLUSTER_NAME \
  --zone $ZONE \
  --num-nodes 3 \
  --machine-type e2-standard-2 \
  --enable-autoscaling \
  --min-nodes 2 \
  --max-nodes 10 \
  --enable-autorepair \
  --enable-autoupgrade \
  --release-channel regular
```

**叢集配置說明**:
- **節點數**: 3 個 (可自動擴展到 2-10)
- **機器類型**: e2-standard-2 (2 vCPU, 8GB RAM)
- **自動修復**: 啟用
- **自動升級**: 啟用
- **發布頻道**: regular (穩定版本)

### 步驟 3: 建立命名空間

```bash
# 建立 flowagent 命名空間
kubectl create namespace $NS --dry-run=client -o yaml | kubectl apply -f -

# 建立 monitoring 命名空間
kubectl create namespace monitoring --dry-run=client -o yaml | kubectl apply -f -

# 驗證
kubectl get namespaces
```

### 步驟 4: 部署應用程式

```bash
# 使用 Kustomize 建置清單
kustomize build cluster/overlays/prod > /tmp/prod-manifests.yaml

# 查看將要部署的資源
kubectl apply -f /tmp/prod-manifests.yaml --dry-run=client

# 實際部署
kubectl apply -f /tmp/prod-manifests.yaml
```

**部署的組件**:
- MongoDB (資料庫)
- Module-A (主服務模組, 2 副本)
- Orchestrator (協調器, LoadBalancer)
- Next.js Frontend (前端應用, LoadBalancer)

### 步驟 5: 等待部署完成

```bash
# 監控 MongoDB 部署
kubectl rollout status deployment/mongodb -n $NS --timeout=5m

# 監控 Module-A 部署
kubectl rollout status deployment/module-a -n $NS --timeout=5m

# 監控 Orchestrator 部署
kubectl rollout status deployment/orchestrator -n $NS --timeout=5m

# 監控 Next.js Frontend 部署
kubectl rollout status deployment/nextjs-frontend -n $NS --timeout=5m
```

### 步驟 6: 部署監控系統 (可選)

```bash
# 部署 Prometheus
kubectl apply -k cluster/overlays/monitoring/

# 驗證監控部署
kubectl get all -n monitoring
```

## 部署驗證

### 自動驗證

使用狀態檢查腳本:
```bash
bash scripts/check_deployment_status.sh
```

腳本會檢查:
- ✅ 命名空間
- ✅ 部署狀態
- ✅ Pod 狀態
- ✅ 服務端點
- ✅ 持久化存儲
- ✅ 資源使用
- ✅ 最近事件

### 手動驗證

#### 1. 檢查所有資源

```bash
kubectl get all -n $NS
```

預期輸出應顯示:
- 4 個 Deployments (全部就緒)
- 4-8 個 Pods (全部 Running)
- 4 個 Services

#### 2. 檢查 Pod 狀態

```bash
kubectl get pods -n $NS -o wide
```

所有 Pods 應該顯示 `Running` 狀態且 `READY` 為 `1/1` 或 `2/2`。

#### 3. 檢查服務端點

```bash
kubectl get svc -n $NS
```

查看 LoadBalancer 服務的 `EXTERNAL-IP`:
- `nextjs-frontend`: 應有外部 IP
- `orchestrator`: 應有外部 IP

#### 4. 獲取訪問 URL

```bash
# Next.js Frontend
echo "Next.js Frontend: http://$(kubectl get svc nextjs-frontend -n $NS -o jsonpath='{.status.loadBalancer.ingress[0].ip}')"

# Orchestrator
echo "Orchestrator: http://$(kubectl get svc orchestrator -n $NS -o jsonpath='{.status.loadBalancer.ingress[0].ip}')"
```

#### 5. 測試應用程式

```bash
# 測試 Next.js Frontend
FRONTEND_IP=$(kubectl get svc nextjs-frontend -n $NS -o jsonpath='{.status.loadBalancer.ingress[0].ip}')
curl -I http://$FRONTEND_IP

# 測試 Orchestrator
ORCH_IP=$(kubectl get svc orchestrator -n $NS -o jsonpath='{.status.loadBalancer.ingress[0].ip}')
curl -I http://$ORCH_IP/health || curl -I http://$ORCH_IP
```

#### 6. 檢查日誌

```bash
# 查看 Module-A 日誌
kubectl logs -f deployment/module-a -n $NS

# 查看 Orchestrator 日誌
kubectl logs -f deployment/orchestrator -n $NS

# 查看 Next.js Frontend 日誌
kubectl logs -f deployment/nextjs-frontend -n $NS
```

## 部署後配置

### 1. 配置域名 (可選)

如果你想使用自定義域名而不是 IP 地址:

#### 1.1 建立 Ingress 資源

創建 `ingress.yaml`:
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: flowagent-ingress
  namespace: flowagent
  annotations:
    kubernetes.io/ingress.class: "gce"
spec:
  rules:
  - host: flowagent.yourdomain.com
    http:
      paths:
      - path: /*
        pathType: ImplementationSpecific
        backend:
          service:
            name: nextjs-frontend
            port:
              number: 3000
```

#### 1.2 應用 Ingress

```bash
kubectl apply -f ingress.yaml
```

#### 1.3 更新 DNS 記錄

獲取 Ingress IP:
```bash
kubectl get ingress flowagent-ingress -n $NS
```

在你的 DNS 提供商處創建 A 記錄指向這個 IP。

### 2. 配置 SSL/TLS 證書

#### 使用 Google Managed Certificate

創建 `managed-cert.yaml`:
```yaml
apiVersion: networking.gke.io/v1
kind: ManagedCertificate
metadata:
  name: flowagent-cert
  namespace: flowagent
spec:
  domains:
    - flowagent.yourdomain.com
```

應用並更新 Ingress:
```bash
kubectl apply -f managed-cert.yaml
```

在 Ingress 中添加 annotation:
```yaml
metadata:
  annotations:
    networking.gke.io/managed-certificates: flowagent-cert
```

### 3. 更新 MongoDB 密碼

**重要**: 生產環境必須更改預設密碼！

```bash
# 生成新密碼
NEW_PASSWORD=$(openssl rand -base64 32)

# 更新 Secret
kubectl create secret generic mongodb-secret \
  --from-literal=mongodb-root-password=$NEW_PASSWORD \
  --from-literal=mongodb-password=$NEW_PASSWORD \
  --namespace=$NS \
  --dry-run=client -o yaml | kubectl apply -f -

# 重啟 MongoDB 以應用新密碼
kubectl rollout restart deployment/mongodb -n $NS
```

**保存密碼**: 將新密碼保存到安全的位置（如密碼管理器或 GCP Secret Manager）。

### 4. 配置備份策略

#### 4.1 MongoDB 備份

創建備份 CronJob:
```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: mongodb-backup
  namespace: flowagent
spec:
  schedule: "0 2 * * *"  # 每天凌晨 2 點
  jobTemplate:
    spec:
      template:
        spec:
          containers:
          - name: backup
            image: mongo:6.0
            command:
            - /bin/sh
            - -c
            - mongodump --host mongodb --out /backup/$(date +%Y%m%d-%H%M%S)
            volumeMounts:
            - name: backup
              mountPath: /backup
          restartPolicy: OnFailure
          volumes:
          - name: backup
            persistentVolumeClaim:
              claimName: mongodb-backup-pvc
```

### 5. 配置監控告警

如果部署了 Prometheus，配置告警規則:

創建 `alerting-rules.yaml`:
```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: prometheus-rules
  namespace: monitoring
data:
  alert.rules: |
    groups:
    - name: flowagent
      rules:
      - alert: HighPodMemory
        expr: container_memory_usage_bytes > 500000000
        for: 5m
        annotations:
          summary: "High memory usage detected"
```

## 故障排除

### 常見問題

#### 1. Pods 無法啟動

**症狀**: Pods 停留在 `Pending` 或 `CrashLoopBackOff` 狀態

**診斷**:
```bash
# 查看 Pod 詳情
kubectl describe pod <pod-name> -n $NS

# 查看 Pod 日誌
kubectl logs <pod-name> -n $NS

# 查看事件
kubectl get events -n $NS --sort-by='.lastTimestamp'
```

**解決方案**:
- 檢查映像是否正確
- 確認節點有足夠資源
- 檢查配置和環境變數

#### 2. 映像拉取失敗

**症狀**: `ErrImagePull` 或 `ImagePullBackOff`

**診斷**:
```bash
kubectl describe pod <pod-name> -n $NS
```

**解決方案**:
```bash
# 確保 GKE 節點有權限訪問 Artifact Registry
gcloud projects add-iam-policy-binding $PROJECT_ID \
  --member="serviceAccount:$(gcloud container clusters describe $CLUSTER_NAME \
    --zone=$ZONE --format='value(nodeConfig.serviceAccount)')" \
  --role="roles/artifactregistry.reader"
```

#### 3. LoadBalancer IP 未分配

**症狀**: Service 的 `EXTERNAL-IP` 顯示 `<pending>`

**診斷**:
```bash
kubectl describe svc <service-name> -n $NS
```

**解決方案**:
- 等待 5-10 分鐘（GCP 需要時間分配 IP）
- 檢查 GCP 配額是否足夠
- 確認防火牆規則允許流量

#### 4. 服務無法連接

**症狀**: 無法訪問應用程式

**診斷**:
```bash
# 測試內部連接
kubectl run test-pod --rm -i --tty --image=busybox -n $NS -- sh
# 在 pod 中執行:
wget -O- http://module-a:8080/health
```

**解決方案**:
- 檢查服務配置
- 確認 Pod 正在運行
- 檢查防火牆規則

## 回滾程序

### 情況 1: 部署失敗需要回滾

```bash
# 查看部署歷史
kubectl rollout history deployment/<deployment-name> -n $NS

# 回滾到上一個版本
kubectl rollout undo deployment/<deployment-name> -n $NS

# 回滾到特定版本
kubectl rollout undo deployment/<deployment-name> -n $NS --to-revision=<revision-number>
```

### 情況 2: 完全刪除部署

```bash
# 刪除所有應用程式資源
kubectl delete -k cluster/overlays/prod/

# 刪除命名空間 (會刪除所有資源)
kubectl delete namespace $NS

# 刪除監控
kubectl delete namespace monitoring
```

**警告**: 刪除命名空間會刪除所有數據！確保已備份重要數據。

### 情況 3: 保留數據的回滾

```bash
# 只刪除 Deployments，保留 PVC
kubectl delete deployment --all -n $NS

# 重新部署
kubectl apply -k cluster/overlays/prod/
```

  --min=2 --max=10 --cpu-percent=80

kubectl apply -k cluster/overlays/prod/
